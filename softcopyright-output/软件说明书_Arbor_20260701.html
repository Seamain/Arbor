Arbor Git 客户端软件 V1.0

一、概述

Arbor Git 客户端软件（以下简称"本软件"）是一款基于 Tauri 2 框架与 React 19 前端技术构建的跨平台原生 Git 图形化客户端。本软件旨在为软件开发人员、项目管理人员及版本控制使用者提供一个界面美观、操作直观、性能高效的 Git 版本控制可视化工具，帮助用户摆脱传统命令行操作 Git 的学习门槛，通过图形化界面完成分支管理、提交历史查看、差异对比、冲突解决、远程仓库协作等常见版本控制任务。

本软件采用 Rust 语言编写后端核心逻辑，通过 Tauri 框架实现与操作系统底层的高效交互，并调用 libgit2（git2 库）实现对 Git 仓库的直接操作，避免了对外部 Git 命令行程序的依赖，从而保证了操作的稳定性与执行效率。前端界面采用 React 19 结合 TypeScript 开发，并使用 Tailwind CSS 与 HeroUI 组件库构建具有玻璃拟态（Glassmorphism）视觉风格的现代化用户界面，支持亮色、暗色及跟随系统三种主题模式，同时内置中文简体、中文繁体与英文三种语言支持，满足不同地区用户的使用需求。

本软件可运行于 macOS、Linux、Windows 三大主流桌面操作系统，安装包体积小、启动速度快、内存占用低，相较于基于 Electron 架构的同类软件具有明显的性能优势。

二、开发目的

随着软件工程团队协作规模的不断扩大，Git 已成为事实上的版本控制标准工具。然而，Git 原生命令行工具学习曲线陡峭，对于分支管理、合并冲突处理等较为复杂的操作，初学者及非专业开发人员往往难以直接掌握，操作失误还可能造成代码丢失等严重后果。市面上已有的图形化 Git 客户端多数基于 Electron 等 Web 容器技术构建，存在启动缓慢、内存占用高、界面风格与操作系统割裂等问题。

本软件的开发目的在于：其一，基于 Tauri 原生框架构建轻量、高效、低资源占用的桌面应用，充分发挥 Rust 语言在系统级编程中的性能与安全优势；其二，通过可视化提交图谱、拖拽式冲突解决器等交互设计，降低用户操作 Git 的认知负担；其三，集成 GitHub、GitLab、Gitee 等主流代码托管平台的 OAuth 授权登录与个人访问令牌管理能力，实现仓库克隆、Pull Request / Merge Request 查看等一体化协作功能，提升开发团队的整体协作效率。

三、运行环境

3.1 硬件环境

  • 处理器：Intel x86_64 或 ARM64（Apple Silicon）架构，主频 1.6GHz 及以上
  • 内存：4GB 及以上
  • 硬盘可用空间：200MB 及以上

3.2 软件环境

  • 操作系统：macOS 11 及以上、Windows 10/11、主流 Linux 发行版（Ubuntu、Fedora、openSUSE 等）
  • 本地已安装 Git（用于部分辅助操作场景）
  • 网络环境：需联网以使用远程仓库克隆、OAuth 登录及在线更新检测等功能

3.3 开发环境

  • 后端语言：Rust（edition 2021）
  • 桌面应用框架：Tauri 2.x
  • 前端语言与框架：TypeScript、React 19
  • 构建工具：Vite 7
  • 样式方案：Tailwind CSS 4、HeroUI 组件库
  • 核心第三方库：git2（libgit2 绑定）、reqwest、tokio、serde、notify（文件系统监听）、llama-cpp-2（本地 AI 推理）

五、软件的技术特点与创新点

5.1 原生高性能架构

本软件基于 Tauri 2 框架构建，前端界面渲染依托操作系统原生 WebView 组件，后端核心逻辑完全由 Rust 编写并编译为原生二进制代码，摒弃了 Electron 等方案内置完整 Chromium 浏览器内核的做法，从而大幅降低安装包体积与运行时内存占用，提升应用启动速度与整体运行效率。

5.2 直接调用 libgit2 的 Git 操作层

不同于部分同类工具通过子进程调用系统 Git 命令行程序解析文本输出的方式，本软件通过 git2 库直接绑定 libgit2 C 语言库，以结构化 API 形式完成仓库状态读取、提交、分支、合并、变基等操作，减少了因命令行输出格式差异导致的解析错误风险，同时具备更高的执行效率与更精细的错误处理能力。

5.3 可视化冲突解决机制

针对合并/变基过程中常见的冲突场景，本软件设计了独立的冲突解析模块，能够自动解析冲突标记（如 <<<<<<< / ======= / >>>>>>>），并提供并排对比与原始编辑两种交互模式，帮助用户以更低的认知成本完成冲突处理。

5.4 统一的多平台身份认证体系

本软件设计并实现了统一的 OAuth 授权与个人访问令牌管理体系，抽象了 GitHub、GitLab、Gitee 等不同代码托管平台的鉴权协议差异，为用户提供一致的登录体验，并将凭证信息进行本地安全存储。

5.5 本地 AI 能力集成

本软件集成了基于 llama.cpp 的本地大语言模型推理引擎，无需依赖云端服务即可在本地完成智能文本生成任务，兼顾了功能智能化与用户数据隐私保护的双重需求。

5.6 实时文件监听与状态同步

本软件基于操作系统原生文件系统事件监听接口，实现对仓库工作区变化的实时感知，并结合防抖（debounce）机制避免频繁刷新造成的性能损耗，保证界面状态与磁盘实际状态的一致性。

六、软件的运行原理与工作流程

本软件启动后，前端 React 应用通过 Tauri 提供的 IPC（进程间通信）机制向 Rust 后端发起指令调用（invoke），后端接收到指令后，通过 git_ops 模块调用 libgit2 完成对应的仓库操作，并将执行结果以 JSON 序列化形式返回前端进行渲染。整体工作流程如下：

  1. 用户通过界面选择或打开本地 Git 仓库，前端调用后端接口读取仓库状态（当前分支、提交历史、工作区变更等）；
  2. 后端 git_ops 模块基于 libgit2 遍历提交对象与引用信息，构建提交图谱数据结构并返回前端进行可视化渲染；
  3. 用户在界面上执行暂存、提交、分支切换、合并等操作时，前端将操作参数通过 IPC 传递至后端，由后端调用相应的 libgit2 接口完成实际的仓库写操作；
  4. 当涉及远程仓库交互（如克隆、拉取 PR/MR 列表）时，oauth 模块负责处理身份鉴权流程，通过 reqwest 异步 HTTP 客户端与 GitHub/GitLab/Gitee 开放 API 通信；
  5. notify 文件监听模块在后台持续监控仓库目录的文件系统事件，一旦检测到外部变更（如通过命令行操作仓库），即通知前端刷新界面状态；
  6. 本地 AI 模块在用户触发智能生成场景时，加载本地量化语言模型进行推理，并将生成结果返回前端展示。

八、开发过程与团队分工

本软件由开发者独立设计、开发与测试完成，开发过程涵盖需求分析、架构设计、前后端功能实现、多平台适配、界面视觉设计以及版本迭代优化等环节。软件采用模块化架构进行组织：后端划分为 Git 操作模块（git_ops）、身份鉴权模块（oauth）、本地 AI 模块（local_ai）与应用生命周期管理模块（lib）；前端划分为主应用容器（App）、提交图谱组件、设置与偏好管理组件、克隆与远程协作组件、冲突处理组件以及国际化资源模块，各模块职责清晰、耦合度低，便于后续功能扩展与维护。
